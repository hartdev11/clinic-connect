# สเปกระบบคลินิกหลายสาขา (Franchise Model)

**สถานะ:** Phase 1 + Phase 2 + Enterprise (ยืนยันที่อยู่/โทร, ตรวจสอบสาขาหลัก, บล็อกซื้อ, Finance รวม)  
**อ้างอิง:** หลักการสถาปัตยกรรมใน [ENTERPRISE-FRANCHISE-LICENSE-ARCHITECTURE.md](./ENTERPRISE-FRANCHISE-LICENSE-ARCHITECTURE.md)

### สิ่งที่ implement แล้ว
- **Types & schema:** `Organization` + clinic_type, franchise_role, invite_code, branch_number, address_phone_verified_at; `Branch` + branch_type; collections: `franchise_join_requests`, `verification_tokens`, `main_branch_registry`
- **Registration:** คลินิกเดี่ยว / สาขาหลัก (ต้อง plan multi_branch หรือ enterprise; เลือกตรวจสอบเจ้าของผ่าน API/registry) / สาขาย่อย (กรอกรหัสสาขาหลัก → ส่งคำขอ → สาขาหลักอนุมัติ)
- **API:** POST `/api/auth/register`, POST `/api/auth/franchise-approve`, GET `/api/clinic/franchise/requests`; POST `/api/clinic/verification/send-address-phone`, GET `/api/public/verify-address-phone?token=...`
- **Session & Context:** JWT มี franchise_role, franchise_main_org_id; context franchise.isFranchiseMain, isFranchiseSub
- **UI:** หน้า register, หน้า คำขอเข้าร่วมแฟรนไชส์, หน้า /verify-address-phone (redirect)
- **Dashboard/Insights/Finance แบบรวม:** สาขาหลักเห็นข้อมูลรวมทุกสาขา (รวม Finance + Executive Brief)
- **สาขาย่อยซื้อก่อนสาขาหลัก:** ติ๊กที่หน้าแพ็คเกจ → คำขอ + แจ้งสาขาหลัก → อนุมัติได้ Key + รหัสชั่วคราว; **บล็อก:** ถ้าสาขาหลักมี subscription แล้ว สาขาย่อยซื้อไม่ได้ (คืน 400)
- **Enterprise เพิ่ม:** ยืนยันที่อยู่/โทร (feature flag + อีเมลลิงก์); ตรวจสอบเจ้าของสาขาหลัก (external API หรือ Firestore registry)

เอกสารนี้อธิบายการออกแบบระบบคลินิกที่มีหลายสาขา (Franchise Model) โดยเน้น:
- การจัดการสาขาหลัก (Main Branch) และสาขาย่อย (Franchisee)
- การยืนยันตัวตนอย่างเข้มงวด
- การตรวจสอบสิทธิ์การซื้อแพ็คเกจ
- การจำกัดการเข้าถึงข้อมูลเฉพาะสาขาหลัก (และตามบทบาท)
- ความปลอดภัยสูง และ UX ที่ไม่ยุ่งยากสำหรับทั้งเจ้าของคลินิกหลักและสาขาย่อย
- ระดับ Enterprise และขยายได้ในระยะยาว

---

## 1. การลงทะเบียนสำหรับคลินิก (Registration Process)

### 1.1 กรณี "คลินิกเดี่ยว"

| ขั้นตอน | รายละเอียด |
|--------|-------------|
| **หน้าลงทะเบียน** | ผู้ใช้เลือกประเภทคลินิกเป็น **"คลินิกเดี่ยว"** |
| **กรอกข้อมูล** | ชื่อคลินิก, ที่อยู่, หมายเลขโทรศัพท์, อีเมล์ |
| **ยืนยันความถูกต้อง** | ระบบส่งอีเมล์เพื่อยืนยันการลงทะเบียน (ตรวจสอบความถูกต้องของที่อยู่/หมายเลขโทรศัพท์ตามที่ออกแบบ) |
| **หลังยืนยัน** | ผู้ใช้ได้รับ **key** สำหรับการเข้าระบบ |

### 1.2 กรณี "คลินิกที่มีแฟรนไชส์"

#### 1.2.1 หน้าลงทะเบียน

- ผู้ใช้เลือกประเภทคลินิกเป็น **"คลินิกที่มีแฟรนไชส์"**
- ผู้ใช้เลือกประเภทสาขา: **"สาขาหลัก"** หรือ **"สาขาย่อย"**

#### 1.2.2 หากเลือก "สาขาหลัก"

| ขั้นตอน | รายละเอียด |
|--------|-------------|
| **กรอกข้อมูลสาขาหลัก** | ชื่อคลินิก, หมายเลขสาขา, ที่อยู่, เบอร์โทรศัพท์, อีเมล์ |
| **ตรวจสอบความเป็นเจ้าของ** | ระบบตรวจสอบว่าเป็นเจ้าของคลินิกหลักจริงหรือไม่ (เช่น ผ่านหมายเลขสาขา/รหัส หรือฐานข้อมูลที่ตรวจสอบเจ้าของสาขาหลัก) |
| **ผลการตรวจสอบ** | หากเป็นสาขาหลักที่ถูกต้อง → ระบบอนุญาตให้กรอกข้อมูลเพิ่มเติมและดำเนินการยืนยัน |

#### 1.2.3 หากเลือก "สาขาย่อย"

| ขั้นตอน | รายละเอียด |
|--------|-------------|
| **กรอกข้อมูลสาขาย่อย** | ชื่อสาขา, ที่อยู่, เบอร์โทรศัพท์ |
| **รหัสสาขาหลัก** | ระบบบังคับให้กรอก **รหัสสาขาหลัก** ที่สาขานั้นเชื่อมโยงอยู่ |
| **ส่งคำขอยืนยัน** | ระบบส่งคำขอไปยังเจ้าของสาขาหลักเพื่อยืนยันการเป็นแฟรนไชส์จริง — **เจ้าของสาขาหลักต้องยอมรับคำขอ** ก่อนจึงจะถือว่าสมบูรณ์ |

### 1.3 ยืนยันตัวตนและข้อมูล

- ระบบต้องมีการยืนยัน **อีเมล์** และ **รหัส (key)** ที่ส่งไปเพื่อยืนยันการลงทะเบียน
- **Key ต้องไม่ซ้ำกัน** ระหว่างสาขาหลักและสาขาย่อย เพื่อป้องกันการแสดงข้อมูลของสาขาหลักให้กับสาขาย่อย

---

## 2. การซื้อแพ็คเกจ

### 2.1 การซื้อแพ็คเกจโดยเจ้าของสาขาหลัก

- เจ้าของสาขาหลักซื้อ **แพ็คเกจหลัก** (เช่น Premium Package) และได้รับ **key** ที่ใช้ในการลงทะเบียน
- เมื่อเจ้าของสาขาหลักซื้อแพ็คเกจแล้ว **สาขาย่อยจะไม่สามารถซื้อแพ็คเกจใหม่ได้อีก** — ระบบต้องป้องกันการซ้ำซ้อน

### 2.2 กรณีสาขาย่อยมาซื้อก่อนเจ้าของคลินิก

| ขั้นตอน | รายละเอียด |
|--------|-------------|
| **สาขาย่อยซื้อแพ็คเกจ** | สาขาย่อยสามารถซื้อแพ็คเกจได้ แต่ระบบจะ **ส่งคำขอ** ให้เจ้าของสาขาหลักเพื่ออนุมัติ |
| **เก็บข้อมูลและส่งคำขอ** | หลังสาขาย่อยซื้อ แพ็คเกจแล้ว ระบบเก็บข้อมูลสาขาย่อย และส่งคำขอให้เจ้าของสาขาหลัก **ยืนยัน** |
| **อนุมัติโดยสาขาหลัก** | เจ้าของสาขาหลักต้อง **อนุมัติผ่านระบบ** ก่อนที่สาขาย่อยจะได้รับการยืนยันและใช้บริการได้ |
| **เชื่อมโยง key** | เมื่อสาขาหลักอนุมัติแล้ว **key ของสาขาย่อย** จะเชื่อมโยงกับ **key ของสาขาหลัก** |

---

## 3. การจัดการข้อมูลสำหรับสาขาหลักและสาขาย่อย

### 3.1 ข้อมูลที่สามารถเข้าถึงได้

| บทบาท | สิทธิ์การเข้าถึง |
|-------|------------------|
| **เจ้าของสาขาหลัก** | ดูข้อมูลของ **ทุกสาขาย่อย** (สาขาที่เชื่อมโยง): รายได้, จำนวนลูกค้า, การทำงานของสาขา; ดู **สรุปรายงานรวมของทุกคลินิก**, การวิเคราะห์รายได้, การจัดการโปรโมชัน และฟีเจอร์ที่เกี่ยวกับคลินิกทั้งหมด |
| **เจ้าของสาขาย่อย** | ดูข้อมูล **เฉพาะสาขาของตนเองเท่านั้น**: รายได้จากการจอง, ลูกค้าในสาขานั้น, การทำงานในสาขา; **ไม่สามารถดูข้อมูลสาขาอื่นได้** — เฉพาะเจ้าของสาขาหลักที่ดูข้อมูลได้ครบถ้วน |

### 3.2 การจัดการการเข้าถึงข้อมูลด้วย Key

- **Key สาขาหลัก:** มีสิทธิ์เข้าถึงข้อมูลทั้งหมดของทุกสาขาย่อย
- **Key สาขาย่อย:** มีสิทธิ์เข้าถึงข้อมูลเฉพาะของสาขาเดียว
- **การตรวจสอบ Key:** ทุกครั้งที่ล็อกอินหรือเข้าใช้งาน ระบบต้องตรวจสอบว่า key ที่ใช้ถูกต้องและเป็นของสาขาหลักหรือสาขาย่อยที่ตรงกับ session

## 4. หน้า Dashboard สำหรับเจ้าของสาขาหลัก

### 4.1 ข้อมูลที่แสดงในแดชบอร์ด

**สรุปการดำเนินงานของทุกคลินิก**

- รายได้รวมจากทุกสาขา
- ลูกค้าที่มาจากทุกสาขา
- การจอง และโปรโมชัน ที่กำลังดำเนินการในแต่ละสาขา
- ประสิทธิภาพการทำงานของแต่ละสาขา
- การแจ้งเตือน และสรุปผลจากแต่ละสาขา

**การตั้งค่าและการบริหารจัดการ**

- เจ้าของสาขาหลักสามารถ **เชิญสาขาใหม่ (สาขาย่อย)** มาเชื่อมโยงในระบบได้
- จัดการ **การอัปเกรดแพ็คเกจ** ของสาขาย่อย
- ระบบมี **การแจ้งเตือน** เมื่อสาขาย่อยทำการซื้อแพ็คเกจ

### 4.2 การจัดการการเข้าถึงข้อมูล

- ระบบต้องตรวจสอบสิทธิ์ว่า **เจ้าของสาขาหลัก** หรือ **เจ้าของสาขาย่อย** เข้าถึงได้เฉพาะข้อมูลที่เกี่ยวข้องกับบทบาทของตนเท่านั้น

---

## 5. การเชิญสาขาย่อยเข้าร่วมระบบหลังจากซื้อแพ็คเกจ

### 5.1 กรณีเจ้าของสาขาหลักเชิญสาขาย่อย

- หลังจากเจ้าของสาขาหลักซื้อแพ็คเกจแล้ว สามารถ **เชิญสาขาย่อย (Franchisee)** เข้ามาร่วมได้
- ระบบออก **key ใหม่** สำหรับสาขาย่อยที่เชื่อมโยงกับสาขาหลัก
- สาขาย่อยเข้าระบบด้วย **key ของตัวเอง** และเห็นข้อมูลเฉพาะของสาขาตนเอง

### 5.2 การตรวจสอบการเข้าถึงข้อมูล

- ระบบตรวจสอบสิทธิ์ **key** ที่ใช้ในการเข้าระบบทุกครั้ง เพื่อ **แยกข้อมูล** ระหว่างสาขาหลักและสาขาย่อย

---

## 6. ความปลอดภัย (Security)

- **การยืนยันตัวตน:** ผ่านอีเมล์, OTP หรือ 2FA ตามที่ออกแบบ
- **การตรวจสอบและบันทึก:** บันทึกทุกครั้งที่ key ถูกใช้ในการเข้าระบบ (audit)
- **การจำกัดการเข้าถึง:** จำกัดการเข้าถึงข้อมูลตามสิทธิ์ของแต่ละ key (และบทบาทใน RBAC)

## วิธีใช้ (How to Use)

### คลินิกเดี่ยว (Single)
1. ไป **หน้าแพ็คเกจ** (/packages) → เลือกแพ็คเกจ → กรอกอีเมล → ยืนยันและรับอีเมล
2. คลิกลิงก์ในอีเมลเพื่อ **ยืนยันอีเมล** → ได้ **License Key**
3. ไป **สมัครคลินิก** (/register) → เลือก **คลินิกเดี่ยว** → กรอก License Key, ชื่อคลินิก, เบอร์, อีเมล, รหัสผ่าน → สมัคร
4. ไป **เข้าสู่ระบบ** (/login) → กรอก License Key + อีเมล + รหัสผ่าน

### สาขาหลัก (Franchise Main)
1. ซื้อแพ็คเกจ **Multi-Branch หรือ Enterprise** ที่หน้าแพ็คเกจ (เหมือนคลินิกเดี่ยว) → ยืนยันอีเมล → ได้ License Key
2. ไป **สมัครคลินิก** (/register) → เลือก **คลินิกที่มีแฟรนไชส์** → เลือก **สาขาหลัก**
3. กรอก License Key, ชื่อคลินิก, หมายเลขสาขา (ถ้ามี), เบอร์, อีเมล, รหัสผ่าน → สมัคร
4. หลังสมัครสำเร็จ ระบบจะแสดง **รหัสเชิญ** (รูปแบบ INV-xxxxxx-xxxxxx) — **เก็บรหัสนี้ไว้** ให้สาขาย่อยใช้ตอนลงทะเบียนหรือตอนเลือกแพ็คเกจ
5. เข้าสู่ระบบ → ในเมนูซ้ายจะมี **แฟรนไชส์ → คำขอเข้าร่วม** (เฉพาะสาขาหลัก) ใช้ดูและอนุมัติ/ปฏิเสธคำขอจากสาขาย่อย
6. **Dashboard / Insights** จะแสดงข้อมูล **รวมทุกสาขา** (สาขาหลัก + สาขาย่อยที่อนุมัติแล้ว) อัตโนมัติ

### สาขาย่อย — วิธีที่ 1: สาขาหลักอนุมัติก่อน (สมัครแล้วส่งคำขอ)
1. ขอ **รหัสเชิญสาขาหลัก** จากเจ้าของสาขาหลัก (ได้จากขั้นตอนสาขาหลักข้อ 4)
2. ไป **สมัครคลินิก** (/register) → เลือก **คลินิกที่มีแฟรนไชส์** → เลือก **สาขาย่อย**
3. กรอก **รหัสสาขาหลัก**, ชื่อสาขา, ที่อยู่, เบอร์, อีเมล, รหัสผ่าน → กด **ส่งคำขอเข้าร่วม**
4. รอเจ้าของสาขาหลัก **อนุมัติ** ที่ แฟรนไชส์ → คำขอเข้าร่วม
5. หลังอนุมัติ จะได้รับอีเมลพร้อม **License Key** → ใช้ **License Key + อีเมล + รหัสผ่านที่สมัคร** เข้าสู่ระบบ (/login)

### สาขาย่อย — วิธีที่ 2: สาขาย่อยซื้อแพ็คเกจก่อน (แล้วให้สาขาหลักอนุมัติ)
1. ขอ **รหัสเชิญสาขาหลัก** จากเจ้าของสาขาหลัก
2. ไป **หน้าแพ็คเกจ** (/packages) → เลือกแพ็คเกจ → **ติ๊ก "ลงทะเบียนเป็นสาขาย่อย (มีรหัสสาขาหลัก)"**
3. กรอก **รหัสสาขาหลัก**, **ชื่อสาขา**, ที่อยู่/เบอร์ (ถ้ามี), อีเมล → ยืนยัน
4. ระบบจะ **แจ้งอีเมลไปที่สาขาหลัก** — เจ้าของสาขาหลักไปที่ แฟรนไชส์ → คำขอเข้าร่วม → **อนุมัติ**
5. หลังอนุมัติ สาขาย่อยจะได้รับอีเมลพร้อม **License Key** และ **รหัสผ่านชั่วคราว** → ใช้เข้าสู่ระบบ (/login) แล้วควรเปลี่ยนรหัสผ่าน

### สาขาหลัก: อนุมัติคำขอ
1. Login เป็นสาขาหลัก
2. เมนูซ้าย → **แฟรนไชส์** → **คำขอเข้าร่วม**
3. จะเห็นรายการคำขอที่รออนุมัติ (ชื่อสาขา, อีเมล, เบอร์, ที่อยู่)
4. กด **อนุมัติ** หรือ **ปฏิเสธ** — อนุมัติแล้วสาขาย่อยจะได้อีเมลพร้อม License Key (และรหัสชั่วคราวถ้าเป็นกรณีซื้อแพ็คเกจก่อน)

## การออกแบบระดับ Enterprise (ที่ทำเพิ่ม)

### 1. ยืนยันที่อยู่/โทรศัพท์
- **Feature flag:** `ADDRESS_PHONE_VERIFICATION_ENABLED=true`
- **Organization:** `address_phone_verified_at`, `address_phone_verification_sent_at` (ISO)
- **Collection:** `verification_tokens` (doc id = token), หมดอายุ 24 ชม.
- **API:** POST `/api/clinic/verification/send-address-phone` (login, owner/manager) → ส่งอีเมล; GET `/api/public/verify-address-phone?token=...` → อัปเดต org, consume token, audit `address_phone_verified`
- **หน้า:** `/verify-address-phone?token=...` redirect ไป API

### 2. ตรวจสอบเจ้าของสาขาหลัก
- **สถานะ:** พักไว้ก่อน — รอลูกค้าให้ข้อมูลแบบละเอียดว่าต้องการให้ระบบนี้ทำงานแบบไหน ก่อนจะกำหนด spec และเปิดใช้จริง
- **ตำแหน่งในระบบ:** หน้า **ลงทะเบียน** (`/register`) เมื่อเลือก **คลินิกที่มีแฟรนไชส์ → สาขาหลัก** จะมีช่อง **「เลขที่ใช้ยืนยันสาขาหลัก」** + ปุ่ม **ตรวจสอบ** (บังคับเมื่อเปิดใช้การตรวจสอบผ่าน env)
- **การตรวจสอบจริง:** เปิดเมื่อตั้งค่า `HSS_VERIFY_*` / `EXTERNAL_MAIN_BRANCH_VERIFY_URL` / `MAIN_BRANCH_REGISTRY_REQUIRED=true` (ดู `src/lib/feature-flags.ts` — `isMainBranchVerificationRequired()`)
- **Collection:** `main_branch_registry` — doc id = หมายเลขสาขา (normalized), fields: `verified_at`, `source` (manual|external), `external_id`
- **Env:** `EXTERNAL_MAIN_BRANCH_VERIFY_URL` — POST `{ branch_number, org_name }` คาด `{ ok: boolean }`; `MAIN_BRANCH_REGISTRY_REQUIRED=true` — ต้องมี doc ใน registry ถึงจะลงทะเบียนสาขาหลักได้
- **Lib:** `verifyMainBranchOwnership(branchNumber, orgName)` ใน `src/lib/main-branch-verification.ts`; `setMainBranchRegistryVerified()` สำหรับ admin/ระบบภายนอก

**คลินิกความงามในไทย — ตรวจสอบจากเว็บกรม สบส. จริง**
- **เลขที่ใบอนุญาตสถานพยาบาลเอกชน 11 หลัก** จาก กรมสนับสนุนบริการสุขภาพ (กรม สบส.) ตรวจสอบได้ที่ [hosp.hss.moph.go.th](https://hosp.hss.moph.go.th/) หรือ [privatehospital.hss.moph.go.th](https://privatehospital.hss.moph.go.th/)
- **วิธีให้ระบบตรวจจากเว็บจริง:** ตั้งค่า env `HSS_VERIFY_ENABLED=true` และ `HSS_VERIFY_SEARCH_URL` ชี้ไปยัง URL หน้าค้นหา/ผลลัพธ์ของเว็บกรม สบส. ที่เมื่อใส่เลขใบอนุญาตแล้วจะแสดงผลในหน้า (ใช้ placeholder `{license}` ใน URL แทนเลข 11 หลัก เช่น `https://hosp.hss.moph.go.th/result.php?no={license}`) — วิธีหา URL ที่ถูก: เปิดเว็บกรม สบส. → ค้นหาด้วยเลขใบอนุญาต → ดูที่ Address bar หรือ DevTools (Network) ว่า request เป็น GET/POST ไปที่ URL อะไร แล้วนำมาใส่โดยแทนเลขด้วย `{license}`
- **ทางเลือกอื่น:** (1) **Allowlist:** Admin ตรวจจากเว็บกรม สบส. แล้วเพิ่มเลขลง Firestore `main_branch_registry` (2) **API ภายนอก:** ตั้ง `EXTERNAL_MAIN_BRANCH_VERIFY_URL` ให้ POST `{ branch_number, org_name }` คืน `{ ok: true }`

### 3. บล็อกสาขาย่อยซื้อเมื่อสาขาหลักมี subscription
- ใน POST `/api/public/purchase-record` เมื่อ `franchiseSub` + `mainBranchCode`: หลัง resolve mainOrgId เรียก `getSubscriptionByOrgId(mainOrgId)`; ถ้า `status === 'active'` คืน 400 ข้อความให้ใช้สิทธิ์จากสาขาหลัก

### 4. Finance แบบรวมหลาย org
- **Lib:** `getExecutiveFinanceDataForOrgs(orgIds, period, periodValue)` ใน `src/lib/financial-data/executive.ts` — รวม revenue, byService/byDoctor/byBranch/byChannel ตาม key, revenueTrends12Months รวมตาม yearMonth, financialHealth เป็นค่าเฉลี่ย
- **Routes:** GET `/api/clinic/finance`, GET `/api/clinic/finance/executive-brief` ใช้ `getOrgIdsForSession(session)`; ถ้า `orgIds.length > 1` เรียก `getExecutiveFinanceDataForOrgs` แทน `getExecutiveFinanceData`
